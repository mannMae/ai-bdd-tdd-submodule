# BDD Workflow & Design Guidelines

모든 작업은 아래의 단계별 순서를 엄격히 따르며, 단계의 누락이나 역행은 허용되지 않습니다. 특히 AI Agent는 지정된 구간에서 반드시 멈춰야 합니다.

## 1. 사전 준비: 프로젝트 구조 및 환경 동기화 (0단계)
AI Agent는 작업을 시작하기 전에 반드시 부모 프로젝트 루트(또는 `.agents-local/`)에 있는 **`PROJECT_MAP.md`** 파일을 먼저 로드하고 파싱해야 합니다.
- 프로젝트의 물리적 디렉토리 레이아웃(FE, BE, AI 모듈의 실제 경로)을 파악합니다.
- 사용해야 하는 테스트 도구, 프레임워크 및 테스트 실행 명령어(Recipes)를 로드하여 작업 중 엉뚱한 경로에 파일을 생성하거나 틀린 테스트 명령어를 실행하는 실수를 방지합니다.

---

## 2. 단계별 작업 순서 (BDD-TDD 핵심 프로세스)
모든 기능 개발은 다음 순서에 따라 분기하여 한 단계도 빠짐없이 철저히 준수하여 진행합니다.

1. **유저플로우 (User Flow) 작성**: 요구사항을 기반으로 백엔드/프론트엔드/사용자 간의 상호작용 및 시퀀스를 다이어그램(`*_flow.md`)으로 설계합니다.
2. **시나리오 (Scenario) 작성**: 유저플로우의 모든 흐름과 예외 분기를 담은 거킨 시나리오(`.feature`)를 작성합니다.
3. **아키텍처 분기 진행**:
   - **프론트엔드 (Frontend)**:
     1. **통합 테스트 작성 [Red]**: 사용자 시나리오(`.feature`) 흐름을 직접 검증하는 테스트 코드를 작성하고 협력 유닛과 동작 규칙 명세 기입.
     2. **유닛 테스트 작성 [Red]**: 명세된 규칙별 단위 테스트 작성.
     3. **유닛 코드 구현 [Green]**: 단위 테스트 통과 및 최종 통합 테스트 통과를 위한 프로덕션 코드 구현.
   - **백엔드 (Backend)**:
     1. **백엔드 시나리오 (Backend Scenario) 작성**: 사용자 시나리오(`.feature`)에서 백엔드가 관장하는 영역을 API 요청, 페이로드 명세, DB 상태, 상태 코드 등으로 구체화한 거킨 시나리오(`*_backend.feature`)를 작성합니다.
     2. **통합 테스트 작성 [Red]**: 백엔드 시나리오(`*_backend.feature`)를 검증하는 통합 API 테스트(pytest 등) 코드를 작성하고 협력 유닛(라우터, 서비스, 모델 등)과 동작 규칙 명세 기입.
     3. **유닛 테스트 작성 [Red]**: 명세된 규칙별 단위 테스트 작성.
     4. **유닛 코드 구현 [Green]**: 단위 테스트 통과 및 최종 통합 테스트 통과를 위한 프로덕션 코드 구현.
4. **🛑 승인 게이트 및 커밋 (Stop & Wait & Commit)**:
   - 설계/테스트(통합 및 유닛 테스트) 작성 단계가 끝나면 사용자의 승인을 거쳐 기술 RTM(technical_rtm) 자가 채점 및 마스터 백로그(master_backlog) 업데이트를 진행하고, 관련 스펙/코드들을 한데 묶어 논리적 단위로 커밋합니다.

## 3. 🚨 AI 강제 정지(Stop) 및 승인 게이트 (Approval Gate) - 절대 어기지 마십시오!
<SYSTEM_CONSTRAINT>
AI Agent는 다음 행위를 엄격히 금지합니다:
1. 설계 및 테스트 코드(통합/유닛 테스트)를 작성한 뒤, 사용자의 **명시적 승인(예: "승인", "진행해")이 떨어지기 전**에 프로덕션 코드(실제 기능 구현 코드) 작성을 임의로 개시하는 행위.
2. "이어서 진행할까요?" 라고 물어놓고, 사용자의 답변을 기다리지 않고 같은 턴(Turn) 내에서 곧바로 프로덕션 코드를 스스로 작성하기 시작하는 행위.

이 규칙을 어기는 것은 AI로서의 신뢰를 깨는 **치명적인 시스템 실패(Critical Failure)**입니다. 설계/테스트 작성 단계가 종료되면 **반드시 응답 생성을 중단하고 턴(Turn)을 종료**하여 사용자의 승인을 대기하십시오.

*단, 사용자의 명시적인 승인에 의해 프로덕션 코드 구현 단계(Implementation Phase)에 진입한 이후에는, 코드 완성을 위한 점진적 TDD 사이클(테스트 케이스 개별 실행, 최소 코드 수정 및 리팩토링)을 자율적으로 돌 수 있으며, 매 마이크로 루프마다 사용자의 승인을 대기하지 않습니다.*
</SYSTEM_CONSTRAINT>

## 4. 유저플로우(User Flow) 작성 세부 규칙
유저플로우는 사용자의 경험과 동작 흐름을 시각적으로 설계하는 문서입니다.
1. **시퀀스 다이어그램 필수**: 모든 유저플로우 문서에는 백엔드/프론트엔드/사용자 간의 상호작용을 나타내는 시퀀스 다이어그램(Sequence Diagram)을 반드시 작성합니다.
2. **명명 규칙**: 문서의 이름은 반드시 `{기능명}_flow.md` 형식으로 짓습니다.
3. **단일 책임**: 한 기능당 하나의 별도 문서를 생성하여 관리합니다. 여러 기능을 한 문서에 섞지 않습니다.
4. **플로우차트 작성**: 만약 해당 기능이 사용자의 직접적인 조작과 의사결정에 의한 플로우(Interactive Flow)를 포함한다면, 시퀀스 다이어그램뿐만 아니라 플로우차트(Flowchart)도 추가로 작성해야 합니다.

---

## 5. 시나리오(Scenario) 작성 세부 규칙
유저플로우를 기반으로 구체적인 테스트 가능한 시나리오(`.feature` 파일)를 작성합니다.
1. **모든 분기 커버**: 유저플로우 문서에 명시된 모든 흐름과 예외 분기들을 빠짐없이 시나리오로 표현해야 합니다.
2. **영어 키워드 사용**: Gherkin 문법의 핵심 키워드(`Feature`, `Background`, `Scenario`, `Given`, `When`, `Then`, `And`, `But`)는 반드시 영어로 표기합니다. 내용 자체는 한글로 작성하여 이해를 돕습니다.
3. **출처 표기**: 각 시나리오가 구체적으로 어떤 기능과 유저플로우를 다루는지 문서 상단에 알기 쉽게 표기합니다.
4. **동기화**: 유저플로우 문서가 변경되면, 연관된 시나리오(`.feature`)도 반드시 함께 변경하여 최신 상태를 유지합니다.

---

## 6. 커밋(Commit) 규칙
* 기능(Feature)의 구현이 완전히 끝나서 테스트가 통과하면, 작업의 흔적을 남기기 위해 즉시 커밋을 진행해야 합니다. 
* 이때 변경된 문서(기술 매핑 문서, 시나리오 등)와 코드(테스트, 기능코드)는 하나의 커밋으로 논리적으로 묶여야 합니다.
