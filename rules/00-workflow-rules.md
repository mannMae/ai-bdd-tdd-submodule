# 0. 워크플로우(Workflow) 및 AI 행동 규칙

모든 작업은 아래의 5단계 순서를 엄격히 따르며, 단계의 누락이나 역행은 허용되지 않습니다. 특히 AI Agent는 지정된 구간에서 반드시 멈춰야 합니다.

## 1. 단계별 작업 순서 (BDD-TDD 핵심 프로세스)
모든 기능 개발은 다음 순서(`유저플로우 ➔ 시나리오 ➔ 통합테스트 ➔ 유닛테스트 ➔ 유닛 코드`)를 한 단계도 빠짐없이 철저히 준수하여 진행합니다.

1. **유저플로우 (User Flow) 작성**: 요구사항을 기반으로 백엔드/프론트엔드/사용자 간의 상호작용 및 시퀀스를 다이어그램(`*_flow.md`)으로 설계합니다.
2. **시나리오 (Scenario) 작성**: 유저플로우의 모든 흐름과 예외 분기를 담은 거킨 시나리오(`.feature`)를 작성합니다.
3. **통합 테스트 (Integration Test) 작성 [Red]**:
   - 시나리오의 전체 사용자 흐름을 직접 검증하는 통합 테스트 코드를 먼저 작성합니다.
   - **[유닛 및 규칙 명시 필수]**: 통합 테스트 파일 또는 테스트 스위트 상단에 **이 시나리오에서 협력/사용하는 개별 유닛들(컴포넌트, API 훅, Usecase 서비스, VO 등)의 목록**과 **각 유닛이 시나리오 상에서 만족해야 하는 동작 규칙(제약 조건)**을 주석으로 반드시 명시해야 합니다.
4. **유닛 테스트 (Unit Test) 작성 [Red]**:
   - 통합 테스트를 성공시키기 위해 필요한 개별 유닛들의 세부 규격(Specification)을 검증하는 단위 테스트를 작성합니다.
   - **[규칙 기반 테스트]**: 유닛 테스트는 이전 단계 통합 테스트에서 정의된 **개별 유닛의 구체적인 규칙 항목에 대해서만 테스트 케이스를 추가**하며, 임의의 구현 내용을 테스트하지 않습니다.
5. **유닛 코드 (Unit Code / Production Code) 구현 [Green]**:
   - 유닛 테스트에서 추가된 **해당 동작 규칙에 대해서만 프로덕션 코드로 구현**하며, 검증 조건을 충족하여 최종적으로 상위 통합 테스트가 모두 성공하도록 구현합니다.
6. **🛑 승인 게이트 및 커밋 (Stop & Wait & Commit)**:
   - 전체 테스트가 통과하면 사용자의 승인을 거쳐 기술 매핑 문서(RTM) 자가 채점 및 관련 스펙/코드들을 한데 묶어 논리적 단위로 커밋합니다.

## 2. 🚨 AI 강제 정지(Stop) 및 승인 게이트 (Approval Gate) - 절대 어기지 마십시오!
<SYSTEM_CONSTRAINT>
AI Agent는 다음 행위를 엄격히 금지합니다:
1. 테스트 작성 후, 사용자의 **명시적 승인(예: "승인", "진행해")이 떨어지기 전**에 프로덕션 코드(기능 구현 코드)를 작성하는 행위.
2. "이어서 진행할까요?" 라고 물어놓고, 사용자의 답변을 기다리지 않고 같은 턴(Turn) 내에서 곧바로 코드를 스스로 작성하는 행위.

이 규칙을 어기는 것은 AI로서의 신뢰를 깨는 **치명적인 시스템 실패(Critical Failure)**입니다. 
테스트 코드를 작성한 후에는 **반드시 응답 생성을 중단하고 턴(Turn)을 종료**하여 사용자의 응답을 대기하십시오.
</SYSTEM_CONSTRAINT>

## 3. 커밋(Commit) 규칙
*   기능(Feature)의 구현이 완전히 끝나서 테스트가 통과하면, 작업의 흔적을 남기기 위해 즉시 커밋을 진행해야 합니다. 
*   이때 변경된 문서(기술 매핑 문서, 시나리오 등)와 코드(테스트, 기능코드)는 하나의 커밋으로 논리적으로 묶여야 합니다.
