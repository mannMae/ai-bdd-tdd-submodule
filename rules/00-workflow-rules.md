# 0. 워크플로우(Workflow) 및 AI 행동 규칙

모든 작업은 아래의 단계별 순서를 엄격히 따르며, 단계의 누락이나 역행은 허용되지 않습니다. 특히 AI Agent는 지정된 구간에서 반드시 멈춰야 합니다.

## 1. 사전 준비: 프로젝트 구조 및 환경 동기화 (0단계)
AI Agent는 작업을 시작하기 전에 반드시 부모 프로젝트 루트(또는 `.agents-local/`)에 있는 **`PROJECT_MAP.md`** 파일을 먼저 로드하고 파싱해야 합니다.
- 프로젝트의 물리적 디렉토리 레이아웃(FE, BE, AI 모듈의 실제 경로)을 파악합니다.
- 사용해야 하는 테스트 도구, 프레임워크 및 테스트 실행 명령어(Recipes)를 로드하여 작업 중 엉뚱한 경로에 파일을 생성하거나 틀린 테스트 명령어를 실행하는 실수를 방지합니다.

## 2. 단계별 작업 순서 (BDD-TDD 핵심 프로세스)
모든 기능 개발은 다음 순서에 따라 분기하여 한 단계도 빠짐없이 철저히 준수하여 진행합니다.

1. **유저플로우 (User Flow) 작성**: 요구사항을 기반으로 백엔드/프론트엔드/사용자 간의 상호작용 및 시퀀스를 다이어그램(`*_flow.md`)으로 설계합니다.
2. **시나리오 (Scenario) 작성**: 유저플로우의 모든 흐름과 예외 분기를 담은 거킨 시나리오(`.feature`)를 작성합니다.
3. **아키텍처 분기 진행**:
   - **프론트엔드 (Frontend)**:
     1. **통합 테스트 작성 [Red]**: 사용자 시나리오(`.feature`) 흐름을 직접 검증하는 테스트 코드를 작성하고 협력 유닛과 동작 규칙 명세 기입.
     2. **유닛 테스트 작성 [Red]**: 명세된 규칙별 단위 테스트 작성.
     3. **유닛 코드 구현 [Green]**: 단위 테스트 통과 및 최종 통합 테스트 통과를 위한 프로덕션 코드 구현.
   - **백엔드 (Backend)**:
     1. **백엔드 시나리오 (Backend Scenario) 작성**: 사용자 시나리오(`.feature`)에서 백엔드가 관장하는 영역을 API 요청, 페이로드 명세, DB 상태, 상태 코드 등으로 구체화한 거킨 시나리오(`*_backend.feature`)를 작성합니다.
     2. **통합 테스트 작성 [Red]**: 백엔드 시나리오(`*_backend.feature`)를 검증하는 통합 API 테스트(pytest 등) 코드를 작성하고 협력 유닛(라우터, 서비스, 모델 등)과 동작 규칙 명세 기입.
     3. **유닛 테스트 작성 [Red]**: 명세된 규칙별 단위 테스트 작성.
     4. **유닛 코드 구현 [Green]**: 단위 테스트 통과 및 최종 통합 테스트 통과를 위한 프로덕션 코드 구현.
4. **🛑 승인 게이트 및 커밋 (Stop & Wait & Commit)**:
   - 설계/테스트(통합 및 유닛 테스트) 작성 단계가 끝나면 사용자의 승인을 거쳐 기술 매핑 문서(RTM) 자가 채점 및 관련 스펙/코드들을 한데 묶어 논리적 단위로 커밋합니다.

## 3. 🚨 AI 강제 정지(Stop) 및 승인 게이트 (Approval Gate) - 절대 어기지 마십시오!
<SYSTEM_CONSTRAINT>
AI Agent는 다음 행위를 엄격히 금지합니다:
1. 테스트 작성 후, 사용자의 **명시적 승인(예: "승인", "진행해")이 떨어지기 전**에 프로덕션 코드(기능 구현 코드)를 작성하는 행위.
2. "이어서 진행할까요?" 라고 물어놓고, 사용자의 답변을 기다리지 않고 같은 턴(Turn) 내에서 곧바로 코드를 스스로 작성하는 행위.

이 규칙을 어기는 것은 AI로서의 신뢰를 깨는 **치명적인 시스템 실패(Critical Failure)**입니다. 
테스트 코드를 작성한 후에는 **반드시 응답 생성을 중단하고 턴(Turn)을 종료**하여 사용자의 응답을 대기하십시오.
</SYSTEM_CONSTRAINT>

## 4. 커밋(Commit) 규칙
*   기능(Feature)의 구현이 완전히 끝나서 테스트가 통과하면, 작업의 흔적을 남기기 위해 즉시 커밋을 진행해야 합니다. 
*   이때 변경된 문서(기술 매핑 문서, 시나리오 등)와 코드(테스트, 기능코드)는 하나의 커밋으로 논리적으로 묶여야 합니다.
