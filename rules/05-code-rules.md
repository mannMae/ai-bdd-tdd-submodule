# 5. 기능코드(Production Code) 작성 원칙

테스트가 검증하고자 하는 실제 비즈니스 로직과 UI 컴포넌트 구현 원칙입니다.

1. **테스트 우선 및 규칙 한정 구현**:
   - 어떠한 경우에도 반드시 테스트 코드를 우선적으로 작성하고, 그 후에 기능 코드를 작성합니다. (TDD 원칙 준수)
   - 실제 기능 코드는 이전 단계(통합 테스트 상단 명세 ➔ 유닛 테스트)에서 정의된 **동작 규칙에 대해서만 코드로 구현**하며, 규칙에 어긋나거나 명시되지 않은 임의의 비즈니스 로직 및 부가 기능을 프로덕션 코드에 추가하지 않습니다.
2. **모듈화**: 코드는 독립적으로 분리(모듈화)하여, 단위 테스트가 쉽고 다른 곳에서 재사용이 용이하도록 작성합니다.
3. **순수 함수 지향**: 상태를 변형시키거나 부수 효과(Side Effect)를 일으키지 않는 순수 함수(Pure Function) 형태로 최대한 함수를 작성합니다.
4. **🛑 RTM 열(Column) 매핑 기반 코드 폼(Code Form) 의무 적용**:
   - 에이전트는 기능 기술 매핑 문서(`rtm_*.md`)의 테이블 셀에 나열된 모든 소스 코드를 작성할 때, **해당하는 열(Column)의 헤더에 기재된 `[Code Form ID]`와 단위 테스트 최상단 메타데이터의 `적용 코드 폼 (Code Form)` 필드가 지칭하는 물리 템플릿 파일의 구조를 100% 복사 및 상속하여 구현**해야 합니다.
   - 에이전트는 구현 단계(Green)에서 프로덕션 코드를 구현할 때, 반드시 단위 테스트 파일 최상단의 `적용 코드 폼`에 지정된 물리 경로의 파일 구조와 뼈대를 복사하여 내용을 채워 넣어야 합니다. 임의의 아키텍처나 구조 편차(예: Usecase 서비스를 일반 함수나 여러 비즈니스가 섞인 비대 클래스로 작성하는 행위, VO를 가변 dict로 처리하는 행위 등)는 엄격히 금지되며 즉시 **반려(Reject)** 대상이 됩니다.
   - 표준 코드 양식의 원천은 **코드 폼 사전 문서(`rules/code-form-dictionary.md`)**와 **모의 프로젝트 예시 디렉토리(`rules/exams/apps/`)** 내의 소스 파일들을 따릅니다. 코드를 작성하기 전에 반드시 이 영역들을 먼저 로드하여 대조하십시오.
